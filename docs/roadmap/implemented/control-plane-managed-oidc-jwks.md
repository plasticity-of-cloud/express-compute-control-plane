# Security Hardening — Control-Plane-Managed OIDC/JWKS Keys

## Status: Implemented (v1.0.3-rc30)

## Problem

Currently, when a tenant cluster is provisioned, the SA signing keys and cluster CA are **generated by kubeadm** on the EC2 instance during `kubeadm init`. The JWKS (public key set) is only retrievable after the cluster boots and the API server is reachable. The `ecp` CLI then reads the JWKS from the live API (`/.well-known/openid-configuration`) to register the cluster.

This creates several issues:

1. **Reactive registration** — the cluster must be fully up and publicly reachable before it can be registered. A network hiccup or bootstrap race causes registration to fail silently.
2. **No key custody** — the private SA signing key and CA key live only on the EC2 instance. If the disk is lost or the instance is terminated without backup, recovery is impossible without re-registering.
3. **JWKS is not pre-known** — the control plane cannot pre-register the cluster in DynamoDB before the instance boots.
4. **No rotation path** — currently there is no mechanism to rotate the SA signing key and update the DynamoDB JWKS entry atomically.
5. **No provenance** — there is no cryptographic proof that the CA or SA key was issued by the ecp control plane.

## Proposed Solution

**TenantProvisioningService generates all PKI material (CA + SA signing key) before EC2 launch**, stores everything in Secrets Manager, and pre-registers the JWKS in DynamoDB. The EC2 instance is a stateless consumer — it fetches pre-generated key material from Secrets Manager during boot and passes it to kubeadm.

### Trust Hierarchy

```
KMS Asymmetric Key (RSA-2048, non-exportable, shared, $1/mo)
  │
  │  kms:Sign — CloudTrail audited per invocation
  │
  └── signs per-tenant CA certificate (self-signed CA cert)
        │
        │  CA private key stored in Secrets Manager
        │
        └── per-tenant SA signing key pair
              │  (standalone RSA-2048, stored in Secrets Manager)
              │
              └── JWKS derived from SA public key → DynamoDB
```

- **Root**: Shared KMS asymmetric key — HSM-backed, non-exportable, every `Sign` call logged in CloudTrail
- **Per-tenant CA**: Generated by TenantService, CA cert signed by the KMS root, proves control-plane issuance
- **SA signing key**: Generated by TenantService, used by kube-controller-manager for token signing

### kubeadm Support for External Keys

kubeadm fully supports externally-provided keys. Per [Kubernetes certificate management docs](https://kubernetes.io/docs/tasks/administer-cluster/kubeadm/kubeadm-certs/#custom-certificates):

> If a given certificate and private key pair exists before running `kubeadm init`, kubeadm does not overwrite them.

The relevant files kubeadm looks for under `/etc/kubernetes/pki/`:

| File | Purpose |
|------|---------|
| `ca.key` | CA private key — used to sign component certs (kubelet, apiserver, etc.) |
| `ca.crt` | CA certificate — trust anchor for all cluster TLS |
| `sa.key` | Private RSA key — used by kube-controller-manager to sign SA tokens |
| `sa.pub` | Public RSA key — used by kube-apiserver to verify SA tokens; source of the JWKS |

Placing these files before `kubeadm init` runs is sufficient. No kubeadm config changes are required.

## Provisioning Flow

```
CreateTenant (TenantProvisioningService — Lambda)
  │
  ├── 1. Generate per-tenant CA key pair (RSA-2048, Bouncy Castle)
  ├── 2. Create self-signed CA cert, signed by shared KMS key (kms:Sign)
  ├── 3. Generate SA signing key pair (RSA-2048)
  ├── 4. Derive JWKS from SA public key (jose4j)
  ├── 5. Store in Secrets Manager:
  │       • ecp/tenant/<id>/ca-key   (CA private key, PEM)
  │       • ecp/tenant/<id>/ca-crt   (CA cert, PEM, signed by KMS)
  │       • ecp/tenant/<id>/sa-key   (SA signing private key, PEM)
  ├── 6. Pre-register cluster in DynamoDB:
  │       PK=CLUSTER#<tenantId>
  │       jwks=<derived JWKS>
  │       issuer=https://ecp.codriverlabs.ai/clusters/<tenantId>
  │
  └── 7. Provision EC2 via TenantEc2Service
          └── instance fetches keys from Secrets Manager during boot
              └── kubeadm init uses pre-placed keys (no generation)
```

**The EC2 instance generates nothing.** It is a stateless consumer of pre-generated key material. All cryptographic decisions happen centrally in the Lambda.

## EC2 First-Boot Integration

### Changes required in `express-compute` project

#### File: `eks-d-setup/07-install-eks-d.sh`

Currently this script runs `kubeadm init` without pre-placing any PKI files — kubeadm generates everything. The script must be updated to conditionally fetch PKI from Secrets Manager when `OIDC_MODE=managed`.

**Insert at the top of the script (after `source /opt/eks-d/cluster.env`):**

```bash
# --- Fetch control-plane-managed PKI (before kubeadm init) ---
echo "Fetching PKI material from Secrets Manager..."
sudo mkdir -p /etc/kubernetes/pki

aws secretsmanager get-secret-value \
  --secret-id "ecp/tenant/${TENANT_ID}/ca-key" \
  --query SecretString --output text | sudo tee /etc/kubernetes/pki/ca.key > /dev/null

aws secretsmanager get-secret-value \
  --secret-id "ecp/tenant/${TENANT_ID}/ca-crt" \
  --query SecretString --output text | sudo tee /etc/kubernetes/pki/ca.crt > /dev/null

aws secretsmanager get-secret-value \
  --secret-id "ecp/tenant/${TENANT_ID}/sa-key" \
  --query SecretString --output text | sudo tee /etc/kubernetes/pki/sa.key > /dev/null

# Derive SA public key from private key
sudo openssl rsa -in /etc/kubernetes/pki/sa.key -pubout -out /etc/kubernetes/pki/sa.pub

sudo chmod 600 /etc/kubernetes/pki/ca.key /etc/kubernetes/pki/sa.key
sudo chmod 644 /etc/kubernetes/pki/ca.crt /etc/kubernetes/pki/sa.pub
echo "✓ PKI material fetched from Secrets Manager"
```

No `OIDC_MODE` guard is needed — this script only runs on the Golden AMI, which is always managed mode.

**Add `service-account-issuer` to the kubeadm config template (in the `apiServer.extraArgs` section):**

```yaml
apiServer:
  extraArgs:
    authentication-token-webhook-config-file: /etc/kubernetes/aws-iam-authenticator/kubeconfig.yaml
    enable-admission-plugins: NodeRestriction,MutatingAdmissionWebhook,ValidatingAdmissionWebhook
    service-account-issuer: "https://ecp.codriverlabs.ai/clusters/${TENANT_ID}"
```

kubeadm will see existing `ca.key`, `ca.crt`, `sa.key`, `sa.pub` in `/etc/kubernetes/pki/` → skip generation, and use the configured `service-account-issuer` so tokens contain the deterministic issuer URL.

#### File: `cluster.env`

No new variables are required. `TENANT_ID` is already written by TenantEc2Service and used by `progress.sh`, Karpenter, and CloudWatch scripts. The SM paths are derived from it (`ecp/tenant/${TENANT_ID}/*`).

#### Execution order (unchanged)

The orchestrator `setup-eks-d.sh` step sequence remains the same:

```
Step 1: 05-prepare-etcd.sh           ← etcd volume
Step 2: 06-install-aws-iam-authenticator.sh  ← must precede kubeadm
Step 3: 07-install-eks-d.sh           ← PKI fetch + kubeadm init (modified)
Step 4+: CNI, CCM, cert-manager, etc.
```

No new script file is needed. The Secrets Manager fetch is internal to `07-install-eks-d.sh`.

#### kubeadm config version

The current script uses `kubeadm.k8s.io/v1beta3`. The `service-account-issuer` extra arg works with v1beta3 (map-style `extraArgs`). Migration to v1beta4 (list-style) is a separate concern and not required for this feature.

#### IAM permissions

The instance profile (created by `TenantIamService`) already grants `secretsmanager:GetSecretValue` scoped to `ecp/tenant/<id>/*`. No IAM changes are needed on the EC2 side.

## Security Benefits

| Concern | Current | Proposed |
|---------|---------|----------|
| Key custody | Keys live only on EC2 disk | All keys in Secrets Manager, versioned |
| CA provenance | Self-signed by kubeadm, no audit trail | Signed by KMS key, CloudTrail logged |
| Registration timing | After cluster boots (race condition) | Before instance starts |
| Instance state | Stateful (lose disk = lose keys) | Stateless (terminate and re-provision) |
| Key rotation | Not implemented | Rotate in SM → push to instance → update DynamoDB |
| Issuer URL | kubeadm default | Deterministic, control-plane-assigned |
| JWKS availability | Requires live API access | Available immediately from DynamoDB |

## Cost Analysis (at 2,000 tenants/day)

| Option | Monthly cost |
|--------|-------------|
| ACM PCA — per-tenant CA | 2,000 × $50 = **$100,000/month** |
| ACM PCA — shared CA, per-cert billing | $50 + 59,000 × $0.75 = **$44,250/month** |
| KMS asymmetric key (shared, 1 key) + Bouncy Castle | **$1/month** + negligible Sign ops |
| Self-generated, stored in SM only (no KMS signing) | **$0** extra |

The shared KMS key gives HSM-backed custody and full CloudTrail audit at $1/month. ACM PCA is only justified if the customer's compliance framework explicitly requires it (FedRAMP/DoD).

## Key Rotation (Future)

With keys in Secrets Manager, rotation becomes:

1. Generate new SA key pair in TenantService
2. Store new `sa.key` in Secrets Manager (new version)
3. Add new public key to JWKS in DynamoDB (keep old key during transition)
4. Push new `sa.key` to instance via SSM Run Command → restart controller-manager
5. After grace period (token max lifetime), remove old key from DynamoDB JWKS

Kubernetes supports multiple keys in the JWKS simultaneously, so tokens signed with the old key remain valid during the transition window.

## CDK Stack Configuration

### `oidc-mode` CloudFormation Property

`oidc-mode` is a CloudFormation parameter (not just CDK context), making it configurable at deploy time without code changes.

| Value | Behaviour |
|-------|-----------|
| `managed` (default) | Control plane generates CA + SA keys, stores in SM, pre-registers JWKS. KMS signing key provisioned. Rejects self-managed cluster registrations. |
| `self-managed` | Accepts user-provided JWKS/CA on cluster registration. No KMS key provisioned. No EC2 tenant provisioning. |

```bash
# Default — managed mode (EKS-D tenants with Golden AMI)
cdk deploy

# Self-managed mode (k3s, microk8s, customer-managed clusters)
cdk deploy --context oidcMode=self-managed
# or via CFn parameter override:
aws cloudformation deploy --parameter-overrides OidcMode=self-managed
```

### What the CDK stack provisions per mode

| Resource | `managed` | `self-managed` |
|----------|-----------|----------------|
| KMS asymmetric key (`express-compute/control-plane/ca-signing-key`) | ✅ | ❌ |
| IAM grant: `kms:Sign` for tenant-service Lambda | ✅ | ❌ |
| Tenant-service Lambda (EC2 provisioning) | ✅ | ❌ |
| Launch templates, AMI SSM params | ✅ | ❌ |
| Mgmt-service Lambda (cluster/association CRUD) | ✅ | ✅ |
| Credential-service Lambda | ✅ | ✅ |

## CLI Unified `create-cluster` Command

The CLI exposes a single `ecp create-cluster` command. Both modes route through **tenant-service** (Function URL). The tenant-service owns the clusters table and handles both managed provisioning and self-managed registration.

### `--oidc-mode=managed` (default)

Full tenant provisioning: generates PKI via TenantCryptoService, launches EC2 with Golden AMI, pre-registers JWKS.

```bash
ecp create-cluster \
  --name my-tenant \
  --arch arm64 \
  --pricing-model spot
```

**Rejected parameters**: `--jwks-uri`, `--ca-cert`, `--issuer` — these are control-plane-managed and cannot be overridden.

If user passes self-managed parameters in managed mode:
```
Error: --jwks-uri, --ca-cert, and --issuer cannot be specified in managed mode.
       The control plane generates and manages PKI material for managed clusters.
```

### `--oidc-mode=self-managed`

Registers an externally-managed cluster. No EC2 provisioning, no PKI generation. User must provide JWKS and issuer information. Tenant-service stores the cluster record directly in DynamoDB.

```bash
ecp create-cluster \
  --name my-k3s-cluster \
  --oidc-mode self-managed \
  --jwks-uri https://my-cluster.example.com/.well-known/jwks.json \
  --issuer https://my-cluster.example.com \
  --ca-cert /path/to/ca.crt   # optional, for display/audit only
```

**Required parameters in self-managed mode**: `--jwks-uri` or `--jwks-file`, `--issuer`

If user omits required parameters:
```
Error: Self-managed mode requires the following parameters:
         --jwks-uri <url>  or  --jwks-file <path>    (cluster's JWKS endpoint or local file)
         --issuer <url>                               (SA token issuer URL)
       Optional:
         --ca-cert <path>                             (cluster CA certificate for audit)
```

**Server-side enforcement**: If the CDK stack was deployed with `oidc-mode=managed`, the tenant-service rejects self-managed cluster registrations (HTTP 400):
```json
{
  "error": "This control plane is deployed in managed mode. Self-managed cluster registration is not supported."
}
```

### Routing

Both modes call the **tenant-service Function URL** (`POST /clusters`). The request body includes `oidcMode` to signal which path to take:

```json
// Managed
{"clusterName": "my-tenant", "oidcMode": "managed", "arch": "arm64", "ec2PricingModel": "spot"}

// Self-managed
{"clusterName": "my-k3s", "oidcMode": "self-managed", "jwks": "...", "issuer": "https://..."}
```

The mgmt-service `POST /clusters` remains available as a lower-level API (used by `install-ecp-pod-identity.sh --oidc-mode self-managed`) but the CLI routes exclusively through tenant-service.

### Decision matrix

| `--oidc-mode` | Parameters provided | Result |
|---------------|--------------------:|--------|
| `managed` | (none extra) | ✅ Full provisioning: generate PKI → SM → DynamoDB → EC2 |
| `managed` | `--jwks-uri`, `--issuer` | ❌ Reject: cannot override managed PKI |
| `self-managed` | `--jwks-uri`, `--issuer` | ✅ Register cluster record in DynamoDB |
| `self-managed` | (none extra) | ❌ Error: required parameters missing |
| `self-managed` | (server in managed-only mode) | ❌ Server rejects: mode mismatch |

### `delete-cluster` Server-Side Logic

Tenant-service `DELETE /clusters/{name}`:
1. Look up cluster record in DynamoDB
2. If `managed=true` → full teardown: terminate EC2, release EIP, delete IAM role, delete subnets/SG, delete secrets, delete DLM policy, delete DynamoDB records (tenants + clusters tables)
3. If `managed=false` (self-managed) → delete cluster record from DynamoDB only
4. Return 204 on success

This unifies what was previously two separate flows (`delete-tenant` + `deregister-cluster`) into a single operation where the server determines the teardown scope from the stored record.

### Technical notes

**Key generation** — Use Bouncy Castle (`org.bouncycastle:bcpkix-jdk18on`, already compatible with GraalVM native) for RSA key pair generation and X.509 cert construction. The CA cert is self-signed in the sense that it's a root cert, but the signature comes from the KMS key (via `kms:Sign` with `RSASSA_PKCS1_V1_5_SHA_256`).

**Secrets Manager paths** — Consistent with existing SSH key path (`ecp/tenant/<id>/ssh-key`):
- `ecp/tenant/<id>/ca-key`
- `ecp/tenant/<id>/ca-crt`
- `ecp/tenant/<id>/sa-key`

**IAM** — `TenantIamService` already grants `secretsmanager:GetSecretValue` scoped to `ecp/tenant/<id>/*` on the instance profile. No changes needed for the EC2 side. The Lambda needs `kms:Sign` on the shared KMS key (added by CDK).

**Rollback** — On provisioning failure, `ProvisionedResources.rollback()` must delete the Secrets Manager entries and the pre-registered DynamoDB cluster record.

**GraalVM native compatibility** — Bouncy Castle works with GraalVM native if registered via `reflect-config.json`. The tenant-service already builds as native arm64.

### CLI command quick reference (for all guides)

```bash
# Managed cluster (Express Compute — full provisioning)
ecp create-cluster --name my-cluster --arch arm64 --pricing spot --wait
ecp delete-cluster --name my-cluster
ecp stop-cluster --name my-cluster
ecp resume-cluster --name my-cluster

# Self-managed cluster (k3s, microk8s, existing kubeadm)

  --issuer https://my-node:6443 --jwks-file /tmp/jwks.json

# Associations (same for both modes)

  --service-account app-sa --role-arn arn:aws:iam::123456789012:role/app-role
```

- **Default**: KMS-backed shared signing key + per-tenant self-signed CA stored in Secrets Manager. Auditable via CloudTrail, HSM-backed, near-zero cost.
- **Enterprise add-on**: ACM PCA deployed in the customer's own account. Customer bears the PCA cost. Integrate via `aws-privateca-issuer`.
