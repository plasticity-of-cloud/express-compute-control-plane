# Security Hardening â€” Control-Plane-Managed OIDC/JWKS Keys

## Status: Planned

## Problem

Currently, when a tenant cluster is provisioned, the SA signing keys are **generated by kubeadm** on the EC2 instance during `kubeadm init`. The JWKS (public key set) is only retrievable after the cluster boots and the API server is reachable. The `eks-dx` CLI then reads the JWKS from the live API (`/.well-known/openid-configuration`) to register the cluster.

This creates several issues:

1. **Reactive registration** â€” the cluster must be fully up and publicly reachable before it can be registered. A network hiccup or bootstrap race causes registration to fail silently.
2. **No key custody** â€” the private SA signing key lives only on the EC2 instance. If the disk is lost or the instance is terminated without backup, JWKS rotation is impossible without re-registering.
3. **JWKS is not pre-known** â€” the control plane cannot pre-register the cluster in DynamoDB before the instance boots.
4. **No rotation path** â€” currently there is no mechanism to rotate the SA signing key and update the DynamoDB JWKS entry atomically.

## Proposed Solution

**eks-dx-control-plane generates the SA key pair before provisioning**, stores it in Secrets Manager, and passes it to kubeadm via user-data. The JWKS is registered in DynamoDB as part of the `CreateTenant` flow, before the instance even starts.

### kubeadm Support for External SA Keys

kubeadm fully supports externally-provided SA keys. Per [Kubernetes certificate management docs](https://kubernetes.io/docs/tasks/administer-cluster/kubeadm/kubeadm-certs/#custom-certificates):

> If a given certificate and private key pair exists before running `kubeadm init`, kubeadm does not overwrite them.

The relevant files kubeadm looks for under `/etc/kubernetes/pki/`:

| File | Purpose |
|------|---------|
| `sa.key` | Private RSA key â€” used by kube-controller-manager to sign SA tokens |
| `sa.pub` | Public RSA key â€” used by kube-apiserver to verify SA tokens; source of the JWKS |

Placing these files before `kubeadm init` runs is sufficient. No config changes to kubeadm are required.

### OIDC Issuer URL

The SA token `iss` claim is set via `--service-account-issuer` in the kube-apiserver flags, which kubeadm exposes through `ClusterConfiguration.apiServer.extraArgs`:

```yaml
apiVersion: kubeadm.k8s.io/v1beta4
kind: ClusterConfiguration
apiServer:
  extraArgs:
  - name: service-account-issuer
    value: "https://eks-dx.codriverlabs.ai/clusters/my-tenant"
```

This allows eks-dx-control-plane to assign a **deterministic, per-tenant issuer URL** that is set before provisioning and stored in DynamoDB alongside the JWKS.

## Security Benefits

| Concern | Current | Proposed |
|---------|---------|----------|
| Key custody | Key lives only on EC2 disk | Private key in Secrets Manager, versioned |
| Registration timing | After cluster boots (race condition possible) | Before instance starts |
| Key rotation | Not implemented | Rotate in Secrets Manager â†’ push to instance â†’ update DynamoDB JWKS |
| Issuer URL | Derived from k3s default | Deterministic, control-plane-assigned |
| JWKS availability | Requires live API access | Available immediately from DynamoDB |

## Key Rotation (Future)

With keys in Secrets Manager, rotation becomes:

1. Generate new key pair
2. Store new `sa.key` in Secrets Manager (new version)
3. Add new public key to JWKS in DynamoDB (keep old key during transition)
4. Push new `sa.key` to instance via SSM Run Command â†’ restart controller-manager
5. After grace period, remove old key from DynamoDB JWKS

Kubernetes supports multiple keys in the JWKS simultaneously, so tokens signed with the old key remain valid during rotation.

## Implementation Notes

**Key generation** â€” `TenantIamService` or a new `TenantCryptoService` generates the key pair using the Java `KeyPairGenerator` (RSA-2048). The JWKS derivation reuses the existing `jose4j` dependency already in the credential-service.

**Secrets Manager path** â€” `eks-d-xpress/tenant/<tenantId>/sa-signing-key` (consistent with the existing SSH key path `eks-d-xpress/tenant/<id>/ssh-key`).

**user-data change** â€” The first-boot script (documented in `docs/design/tenant/first-boot-script.md`) needs a step to:
```bash
aws secretsmanager get-secret-value \
  --secret-id "eks-d-xpress/tenant/${TENANT_ID}/sa-signing-key" \
  --query SecretString --output text > /etc/kubernetes/pki/sa.key

# Derive public key from private
openssl rsa -in /etc/kubernetes/pki/sa.key -pubout > /etc/kubernetes/pki/sa.pub
chmod 600 /etc/kubernetes/pki/sa.key
```

This must run before `kubeadm init`.

**kubeadm config** â€” Pass a `ClusterConfiguration` file to `kubeadm init --config` to set the deterministic issuer URL:
```yaml
apiVersion: kubeadm.k8s.io/v1beta4
kind: ClusterConfiguration
apiServer:
  extraArgs:
  - name: service-account-issuer
    value: "https://eks-dx.codriverlabs.ai/clusters/${TENANT_ID}"
```

**IAM** â€” `TenantIamService` already grants `secretsmanager:GetSecretValue` scoped to `eks-d-xpress/tenant/*`. The EC2 instance profile needs no changes.

**DynamoDB cluster record** â€” `TenantProvisioningService.createCluster()` (new step, before EC2 launch) should call `mgmt-service POST /clusters` with `jwks` and `issuer` populated. On rollback, `deleteCluster()` removes the pre-registered entry.

## CA Key Custody Strategy

The roadmap doc above covers the SA signing key. The same pattern applies to the cluster CA (`ca.key` / `ca.crt`), which kubeadm also generates on the instance today.

### How EKS handles it

EKS generates a self-signed CA per cluster entirely within AWS's internal control plane. The private key is stored in an AWS-internal HSM-backed secrets service and never exposed. EKS does **not** use ACM Private CA for cluster CAs â€” at 100 clusters that would be $5,000/month in CA fees alone. The CA is self-signed, per-cluster, and the private key never touches the customer's account.

### Our approach: KMS-backed shared signing key + per-tenant self-signed CA

Generate the CA key pair in `TenantProvisioningService` before EC2 launch, sign the CA cert using a shared KMS asymmetric key (RSA-2048, non-exportable), store the CA key material in Secrets Manager at `eks-d-xpress/tenant/<id>/ca`. The first-boot script writes it to `/etc/kubernetes/pki/ca.key` + `ca.crt` before `kubeadm init`.

**Cost at scale (2,000 tenants/day):**

| Option | Monthly cost |
|--------|-------------|
| ACM PCA â€” per-tenant CA | 2,000 Ă— $50 = **$100,000/month** |
| ACM PCA â€” shared CA, per-cert billing | $50 + 59,000 Ă— $0.75 = **$44,250/month** |
| KMS asymmetric key (shared, 1 key) | **$1/month** + negligible signing ops |
| Self-generated, stored in SM only | **$0** extra (SM encryption is KMS under the hood) |

The $1/month shared KMS signing key with Bouncy Castle issuance gives you HSM-backed key custody and full CloudTrail audit (`kms:Sign` logged per issuance) â€” which is what enterprise customers actually care about. ACM PCA is the right answer only if the customer's compliance framework explicitly requires it (e.g., FedRAMP/DoD), in which case they should bring their own CA into their own AWS account.

### Enterprise positioning

- **Default**: KMS-backed shared signing key + per-tenant self-signed CA stored in Secrets Manager. Auditable via CloudTrail, HSM-backed, near-zero cost.
- **Enterprise add-on**: ACM PCA, deployed in the customer's own account. Customer bears the PCA cost. We integrate via `aws-privateca-issuer`.

### Implementation (same pattern as SA key)

```bash
# First-boot, before kubeadm init
aws secretsmanager get-secret-value \
  --secret-id "eks-d-xpress/tenant/${TENANT_ID}/ca-key" \
  --query SecretString --output text > /etc/kubernetes/pki/ca.key

aws secretsmanager get-secret-value \
  --secret-id "eks-d-xpress/tenant/${TENANT_ID}/ca-crt" \
  --query SecretString --output text > /etc/kubernetes/pki/ca.crt

chmod 600 /etc/kubernetes/pki/ca.key
```

kubeadm does not overwrite existing `ca.key`/`ca.crt` files â€” same behaviour as with `sa.key`/`sa.pub`.

## CDK Stack Configuration

The CDK stack will expose a context flag controlling CA management mode. Default is centrally-managed.

### `clusterCaMode` context flag

| Value | Behaviour |
|-------|-----------|
| `centrally-managed` (default) | Control plane generates CA key pair, stores in Secrets Manager, pre-registers JWKS in DynamoDB before EC2 launch. First-boot script fetches and writes `/etc/kubernetes/pki/ca.key` + `ca.crt`. |
| `self-managed` | kubeadm generates the CA on the instance. CLI registers the cluster post-boot by reading JWKS from the live API. Suitable for customer-managed fleets (k3s, microk8s) where the control plane cannot inject user-data. |

```bash
# Deploy with centrally-managed CA (default â€” kubeadm / EKS-D tenants)
cdk deploy

# Deploy with self-managed CA (k3s fleet or customer-managed clusters)
cdk deploy --context clusterCaMode=self-managed
```

### Use case mapping

| Customer scenario | Recommended mode |
|-------------------|-----------------|
| EKS-D tenants provisioned by eks-dx-tenant-service | `centrally-managed` |
| k3s / microk8s fleet on EC2, customer manages kubeadm | `self-managed` |
| Hybrid (some tenant, some customer-managed) | Deploy two stacks with different context values |

### What changes in the stack

In `centrally-managed` mode the stack provisions an additional KMS asymmetric key (`eks-d-xpress/control-plane/ca-signing-key`) used to sign the self-signed CA certs, and grants `eks-dx-tenant-service` Lambda `kms:Sign` on that key. In `self-managed` mode that KMS key and the associated IAM grant are omitted.

The Secrets Manager paths (`eks-d-xpress/tenant/<id>/ca-key`, `eks-d-xpress/tenant/<id>/ca-crt`) and the SM resource policy on the tenant instance role are the same in both modes â€” the difference is only in who writes those secrets (tenant-service vs. the CLI post-boot in self-managed).
